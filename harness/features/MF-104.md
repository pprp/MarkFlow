# MF-104 - System dark mode applies separate light and dark themes automatically

## Notes
Typora documents separate light and dark theme behavior that follows system appearance automatically. This run extends `packages/desktop/src/main/themeManager.ts`, `packages/desktop/src/preload/index.ts`, `packages/shared/src/index.ts`, and `packages/editor/src/App.tsx` so MarkFlow persists independent light/dark theme ids, exposes them to the renderer, and updates the active theme on Electron `nativeTheme` changes; focused desktop and renderer tests were added, but manual OS-level verification is still pending so `passes` remains false. Sources: https://support.typora.io/Dark-Mode/ and https://support.typora.io/. BUG (found 2026-04-16, live-preview test with `prefers-color-scheme: dark`): In dark mode the editor chrome (`.cm-editor` background `rgb(28, 28, 30)`, default text `rgb(255, 255, 255)`) is swapped correctly, but the CodeMirror syntax-highlight token colors are NOT theme-aware — keyword tokens (class `ͼb`, e.g. `function`, `return`) stay at `rgb(119, 0, 136)` (dark purple) and type tokens (class `ͼi`, e.g. `string`) stay at `rgb(0, 136, 85)` (dark green). Both produce very low contrast against the dark background and fail WCAG AA readability for code blocks. Confirmed via `preview_inspect` on the `function greet(...)` starter example. Fix should either (a) ship a dark-palette HighlightStyle switched on appearance, or (b) wire the existing light/dark theme CSS to override the default `@codemirror/language` `defaultHighlightStyle` colors.

## Steps
- Choose different light and dark themes from the app theme controls and relaunch MarkFlow; confirm both theme mappings persist.
- While MarkFlow is running, switch the OS appearance between light and dark and confirm the matching theme applies immediately without restart.
- Return to each appearance mode and confirm the previously selected light and dark theme ids are still preserved.
