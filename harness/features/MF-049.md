# MF-049 - Incremental parsing updates only dirty AST ranges instead of re-parsing the whole document

## Notes
Implemented via `packages/editor/src/editor/decorations/incrementalParse.ts` plus incremental rebuild support in `inlineDecorations.ts`, which maps existing decorations through document changes and only rebuilds dirty whole-line ranges within the viewport. `packages/editor/src/editor/__tests__/incrementalParse.test.tsx` now locks the dirty-range report and untouched-decoration preservation path, and the inline-decoration EOF boundary regression is covered in `packages/editor/src/editor/__tests__/inlineDecorations.test.tsx`. Automated verification passed on 2026-04-13 (`pnpm --filter @markflow/editor test:run -- --grep incremental`, targeted editor tests, `./harness/init.sh --smoke`, `pnpm --filter @markflow/editor build`, `pnpm --filter @markflow/editor lint`). Not tested: the manual 180k-line parse-scope console check and sub-16 ms latency validation on real M-series hardware.

## Steps
- Open a 180k-line document and type a single character mid-document.
- Confirm the editor re-parses only the affected block (measured via a parse-time log).
- Confirm the total keystroke-to-render latency stays under 16 ms on an M-series MacBook.
