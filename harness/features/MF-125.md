# MF-125 - Split view scroll positions are desynced on load

## Notes
Live test confirmed: when entering split view the source pane scrollTop and preview pane scrollTop diverge immediately (e.g. source 1762, preview 3039). The sync logic uses percentage-based math (`scrollTop / (scrollHeight - clientHeight)`), but the source and preview panes have different scroll heights because the source shows raw markdown and the preview shows rendered content. The initial `handleSourceScroll()` fires before the preview pane has fully rendered its layout, so it syncs against incorrect dimensions.

Root cause: `MarkFlowEditor.tsx` lines 1166–1216 — the initial sync call happens synchronously after `previewView` is created, before a layout pass.

## Steps
- Open any multi-section document and scroll the editor to a mid-document position.
- Switch to Split view.
- Observe that the source pane and preview pane display different sections of the document.

## Automated Verification
- Add a test that switches to split view after setting a mid-document scroll position and asserts both panes display content from the same relative document position (within a small tolerance).

## Manual Verification
- Switch to Split view; both panes should show roughly the same part of the document immediately.
- Scroll either pane; the other should follow proportionally without lag or jump.
