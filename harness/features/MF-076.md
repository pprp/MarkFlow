# MF-076 - Paste as plain text shortcut strips rich formatting before insertion

## Notes
Implemented on 2026-04-15 in `packages/editor/src/editor/extensions/smartPaste.ts` by tracking a short-lived Cmd/Ctrl+Shift+V intent window so the next paste uses `text/plain` even when the clipboard also carries HTML, while preserving the default HTML→Markdown smart-paste path and existing image-paste priority. Focused coverage in `packages/editor/src/editor/__tests__/smartPaste.test.ts` now verifies default rich-text conversion, both `Ctrl+Shift+V` and `Cmd+Shift+V` plain-text shortcut paths, one-shot shortcut reset, blur reset, 1-second timeout expiry, and image-paste precedence. Sources: https://support.typora.io/Copy-Paste/ and https://support.typora.io/Shortcut-Keys/.

Updated on 2026-04-19: re-ran the session-start protocol (`pnpm harness:start`, `./harness/init.sh --smoke`) plus the required automated verification (`pnpm --filter @markflow/editor exec vitest run src/editor/__tests__/smartPaste.test.ts`, `pnpm --filter @markflow/editor lint`, `pnpm --filter @markflow/editor build`, and `pnpm harness:verify`) on the current tree, and all of those commands passed again after expanding `smartPaste.test.ts` with the missing macOS shortcut and reset-window coverage. I then built the current-source packaged desktop app with `pnpm desktop:pack`, unpacked it to `/tmp/mf076-markflow-app/MarkFlow.app`, and pushed the manual verification further with a live browser clipboard probe: a real `Microsoft Edge` page copy exposed `«class HTML»` plus plain-text clipboard flavors for `Bold link and code`, and a native `Cmd+V` into the packaged MarkFlow app saved `**Bold** [link](https://example.com/) and \`code\`` to disk, confirming the default webpage paste path still takes the HTML-to-markdown conversion route. The ledger still cannot move to `status=verified` or `passes=true` in this environment, because the full acceptance matrix remains incomplete: `mdfind "kMDItemCFBundleIdentifier == 'com.microsoft.Word'"` still returns no `Microsoft Word.app` bundle, the native `Cmd+Shift+V` desktop automation path did not produce a trustworthy paste event for the webpage case, and the `Code` source probe did not yield stable clipboard evidence. I did not substitute Pages/TextEdit for Word or treat the split automation probes as manual proof, so `harness/feature-ledger.json` remains truthful at `status=ready`, `passes=false`, and `lastVerifiedAt=null`.

## Steps
- Copy rich-text content from an external source.
- Press Cmd/Ctrl+Shift+V in the editor and confirm the inserted text contains no markdown conversion and no HTML residue.
- Repeat with Cmd/Ctrl+V and confirm smart-paste conversion (MF-026) still applies on the default paste path.
