# MF-075 - Copy writes rich clipboard formats and exposes plain text, Markdown, and HTML Code copy actions

## Notes
Implemented with explicit `copy`, `copy-as-markdown`, and `copy-as-html-code` menu actions, a bounded preload clipboard bridge, and a minimal renderer-side markdown selection serializer for bold, emphasis, links, inline code, and paragraph/line-break structure. Automated coverage now asserts default copy writes `text/html` plus rendered plaintext, while the source-oriented actions write markdown text or HTML fragment text through `packages/editor/src/__tests__/App.test.tsx` and `packages/desktop/src/main/menu.test.ts`. Sources: https://support.typora.io/Copy-and-Paste/ and https://support.typora.io/Shortcut-Keys/.

Updated on 2026-04-19: live verification completed against the current-source packaged desktop app built with `pnpm desktop:pack`, extracted to `/tmp/markflow-mf075-app/MarkFlow.app`, and launched under isolated `--user-data-dir=/tmp/mf075-profile-*` state. A CDP probe replaced the starter note with `Before **bold** [link](https://example.com) and \`code\``, selected the sample in Preview mode, and exercised all three `Edit` menu actions against the real system clipboard plus external TextEdit targets. `Copy` wrote rendered plain text `Before bold link and code` and exposed an `«class HTML»` clipboard flavor; pasting that payload into a rich TextEdit document and copying it back yielded `«class RTF »`, while a plain-text TextEdit document preserved only utf8/plain-text data. `Copy as Markdown` pasted the raw markdown source into the plain-text target, and `Copy as HTML Code` pasted the HTML source fragment into the plain-text target. With that live proof plus the listed automated verification, this feature now satisfies its acceptance gate and the ledger can truthfully move to `status=verified`, `passes=true`, and `lastVerifiedAt=2026-04-19T12:51:23Z`.

Updated on 2026-04-23: reopened Typora 1.13 parity gap closed with an explicit `copy-as-plain-text` action. The desktop Edit menu now exposes `Copy as Plain Text`, the shared menu-action union and renderer bridge route it, and the command palette exposes the same action when opened from an editor selection. The renderer writes only the serialized rendered plaintext (`{ text }`) for this action, with no HTML clipboard flavor and no raw markdown markers. Command-palette copy uses a live editor selection snapshot only when the palette was opened from the editor context, preserving the existing non-editor stale-selection guard.

## Steps
- Select text containing bold, a link, and inline code.
- Run the normal `Copy` action and paste into both a rich-text target and a plain-text target; confirm the default clipboard payload carries rendered rich content.
- Run `Copy as Plain Text` and confirm a plain-text paste contains rendered text with markdown markers removed and no HTML flavor.
- Run `Copy as Markdown` and confirm a plain-text paste contains raw markdown source, then run `Copy as HTML Code` and confirm a plain-text paste contains HTML source.

## Manual Verification
- Completed 2026-04-19 in a live packaged Electron session using TextEdit as the external target:
- `Copy` exposed `«class HTML»` on the system clipboard, round-tripped into a rich TextEdit document as `«class RTF »`, and pasted into a plain-text TextEdit document as `Before bold link and code`.

- `Copy as Plain Text` is covered by automated renderer and desktop-menu tests for the reopened gap; no additional manual gate remains because it writes a text-only payload through the existing clipboard bridge.
