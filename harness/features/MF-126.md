# MF-126 - Split view content sync replaces the entire preview document on every keystroke

## Notes
The split-view content sync dispatches `{ from: 0, to: currentContent.length, insert: content }` on every change, replacing the entire preview document. This tears down and recreates all widget decorations (KaTeX math, Mermaid diagrams, table renderers) on every keystroke, causing visual flicker and degrading performance on large documents.

Root cause: `MarkFlowEditor.tsx` lines 1240–1243. The sync should compute a minimal diff and only replace changed ranges, or use CodeMirror's `ChangeSet` API to mirror the source transaction.

Implemented on 2026-04-21 by mirroring source-pane CodeMirror transactions directly into the split preview when both documents share the same start state, and by falling back to a smallest-prefix/suffix diff when syncing external `content` prop changes. The regression test in `packages/editor/src/editor/__tests__/MarkFlowEditor.test.tsx` simulates 100 rapid source edits on a 5,000+ line split document and asserts every preview dispatch replaces/inserts at most one character while staying under the jsdom budget.

Manual verification on 2026-04-21 used Vite + Playwright on Microsoft Edge with a visible KaTeX block in split preview. During 60 rapid source-pane keypresses, the preview math block stayed mounted as the same DOM node, was never removed, accepted the new preview text, and reported no console errors during the probe.

## Steps
- Switch to Split view.
- Open a document containing a math block or Mermaid diagram.
- Type in the source pane and observe the preview pane — rendered widgets flicker or briefly disappear on every keystroke.
- On a large document (> 5 000 lines) notice keypress latency increases.

- Add a benchmark test that measures the time taken to process 100 rapid keystrokes in split view and asserts it stays below the performance budget defined in MF-112.
