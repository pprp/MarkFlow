# MF-050 - Background indexer builds a symbol table for headings and anchors without blocking the UI thread

## Notes
Implemented on 2026-04-15 in `packages/editor/src/editor/indexer.ts`, `packages/editor/src/editor/MarkFlowEditor.tsx`, `packages/editor/src/App.tsx`, and `packages/editor/src/editor/outline.ts`. The background indexer now feeds the live outline and internal-anchor navigation through `symbolTableField` plus `onSymbolTableChange`, performs an immediate off-stack initial scan, keeps subsequent rescans debounced, and drops stale async results when documents switch mid-index. On 2026-04-17 the batching path was first tightened so it scans line batches incrementally instead of rebuilding ever-larger prefix strings, preserves a running scan until the newer debounced restart actually begins, and stops serializing the entire document just to publish symbol-table changes. Later the same day, the remaining 180k-line hot path was reduced again: `indexer.ts` now streams batch lines without a full upfront `split('\n')` and uses a larger `2,000`-line batch size so early outline headings arrive sooner, `MarkFlowEditor.tsx` suppresses parent-echo `onChange` callbacks when external content replacements are already in sync, `App.tsx` skips no-op content/selection writes and stops recomputing full-document total lines for ordinary tabs, and `AppStatusBar.tsx` moves document statistics onto deferred content so render-path full-text scans do not compete with typing. In a further 2026-04-17 follow-up, `App.tsx` now debounces renderer-side recovery-checkpoint IPC by `750ms` so dirty 180k-line tabs no longer serialize their full content across Electron on every keystroke, and `src/__tests__/App.test.tsx` now locks both delayed checkpoint scheduling and rapid-edit coalescing. Regression coverage in `src/editor/__tests__/indexer.test.ts`, `src/editor/__tests__/outline.test.ts`, `src/editor/__tests__/MarkFlowEditor.test.tsx`, and `src/__tests__/App.test.tsx` passed again on 2026-04-17 with the listed editor test command, plus `pnpm --filter @markflow/editor lint`, `pnpm --filter @markflow/editor build`, and `pnpm harness:verify`. The remaining manual/live gate is still unresolved in this environment: I could prove with a focused Electron CDP debug pass that `Input.insertText` mutates the live 180k document after the page is focused, but a clean-profile end-to-end typing probe against unique 180k copies still did not yield a trustworthy first-keystroke latency number, and OS-level GUI/manual interaction remains unavailable here. `passes` therefore remains false and `lastVerifiedAt` stays null until someone can complete the real “type continuously while the document loads” check.
